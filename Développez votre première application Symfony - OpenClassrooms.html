Compétences évaluées

  • Expliquer les éléments fondamentaux du framework Symfony 4

  • Question 1

    Quel est le rôle du contrôleur front?

    Attention, plusieurs réponses sont possibles.

    • Retourner une réponse à partir de la requête d'un utilisateur

    • Démarrer et configurer le Kernel

    • S'assurer que l'on a bien accès à une page

    • Relie une URL à une fonction écrite en PHP

    Le contrôleur front est le point d'entrée de votre application Symfony. C'est ici que l'autoloader de classes de Composer est inclus, et que le Kernel est démarré, qui sera en charge de retourner une réponse à l'utilisateur en fonction de sa requête.

  • Question 2

    Quel est le rôle du router de Symfony?

    Attention, plusieurs réponses sont possibles.

    • Il relie une URL à n'importe quelle fonction PHP.

    • Il relie une URL, mais il faut que ce soit absolument à une action d'un contrôleur Symfony.

    Le router permet d'associer une URL à n'importe quelle fonction appelable : fonction PHP native, méthode statique d'un objet ou encore une action d'un contrôleur Symfony. Mais il faut impérativement qu'un objet de type Response soit retourné, c'est pourquoi on associe généralement l'action d'un contrôleur au router.

  • Question 3

    Quel est le rôle de l’Event Dispatcher de Symfony?

    Attention, plusieurs réponses sont possibles.

    • Il permet à différentes parties indépendantes de votre application de communiquer entre elles.

    • Il permet d'altérer le comportement de l'application à tout moment de son cycle d'exécution.

    • Il introduit un émetteur d'événements que l'on peut utiliser pour propager nos propres événements. 

    • Il n'est utilisable que dans le framework Symfony.

    C'est l'un des composants les plus puissants de l'écosystème Symfony : il peut être utilisé indépendamment du framework et fournit un système qui permet à de multiples parties de votre application de communiquer, sans devenir pour autant "dépendantes" ou encore couplées.

  • Question 4

    Vrai ou faux ? Cochez les affirmations justes :

    Attention, plusieurs réponses sont possibles.

    • Les souscripteurs d'événements (Event Subscribers) sont moins réutilisables.

    • Les écouteurs d'événements (Event Listeners) sont majoritairement utilisés.

    • Il est possible de créer ses propres événements.

    • La priorité des événements doit être comprise entre 0 et 255.

    N'hésitez pas à relire la section correspondant à cette question : bien comprendre l'intérêt de l'Event Dispatcher est vital pour devenir un bon développeur Symfony.

  • Question 5

    Combien d'écouteurs d'événements sont enregistrés pour l’event kernel.exception ?

    Exécutez la commande suivante pour le savoir :  php bin/console debug:event-dispatcher kernel.exception

    • 6

    • 7

    • 8

    • 0

    Cette commande vous permet de bien vérifier que vos événements sont correctement propagés et que les différents types d'écouteurs sont bien enregistrés : si votre écouteur d'événement ne semble pas fonctionner, c'est souvent parce qu'il n'est pas utilisé !

    Prenez donc le réflexe de vérifier d'abord que ce soit le cas.

  • Question 6

    À quoi sert le container de services de Symfony ?

    Attention, plusieurs réponses sont possibles.

    • Il permet de définir la création des différents objets utilisés lors de l'exécution de l'application.

    • Il conserve en un seul endroit les règles de construction d'un objet.

    • Il sert à gérer les webservices externes que l'on utiliserait dans notre application.

    Le container de services est une pièce centrale du framework Symfony ; son rôle essentiel est de construire et gérer vos objets en un seul endroit. Plus vous l'utiliserez, plus vous en comprendrez les avantages en comparaison avec le fait d'instancier ses objets "à la main".

  • Question 7

    Quel est le principal intérêt de l’autoconfiguration ?

    • Elle permet de retrouver les arguments à injecter dans un objet automatiquement.

    • Elle permet d'ajouter automatiquement un tag à des objets selon les interfaces qu'ils implémentent.

    • Elle affecte des valeurs par défaut à certains objets en fonction des interfaces qu'ils implémentent.

    Dans Symfony, de nombreux services ont un comportement particulier en fonction des tags qu'ils ont. Par exemple, une classe qui est une extension Twig doit avoir le tag "twig.extension" pour être chargée dans le service Twig du framework.

    Avec l'autoconfiguration de services, le conteneur de services ajoutera automatiquement le tag si la classe implémente  Twig\Extension\ExtensionInterface.

  • Question 8

    Par défaut, sur Symfony, quels sont les environnements enregistrés ?

    • test  ,  dev  et  prod

    • prod  et  dev

    • dev  ,  prod  et  pre-prod

    • local  ,  pre-prod  et  prod

    Vous pouvez créer autant d'environnements que vous le voulez dans Symfony, qui dépendent de l'organisation de votre entreprise.

    Par défaut, 3 environnements sont disponibles :

    prod : l'environnement de production, avec peu de logs et aucune gestion d'erreurs ;
    dev : l'environnement de développement, avec beaucoup de logs créés et surtout le web profiler qui est activé ;
    test : un environnement particulier pour exécuter vos tests.

    • Faux
